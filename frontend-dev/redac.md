## 2. Problem Statement

### The Researcher's Dilemma

Researchers across every discipline face a deepening paradox: information has never been more abundant, yet actionable, trustworthy insight has never been harder to produce. This creates the **"Data Rich, Insight Poor" (DRIP) syndrome** — a systemic bottleneck that costs billions of hours in wasted human capital annually.

### Three Core Problems

#### Problem 1: Discovery Overload
- Researchers spend **50–80% of their time** on manual literature screening and data extraction
- A typical systematic review requires evaluating **3,000–10,000+ papers** to yield 20–50 usable studies
- PubMed alone adds **4,000+ new papers per day**; ArXiv, SSRN, and domain databases add thousands more
- **Impact:** Delayed research cycles, missed relevant literature, publication bottlenecks

#### Problem 2: Fragmented Workflows
- The average researcher uses **7–12 different tools** for a single research project (search engines, reference managers, PDF readers, writing tools, collaboration platforms)
- No single platform provides end-to-end support from discovery → synthesis → output generation
- Each tool transition introduces friction, data loss risk, and version control problems
- **Impact:** Cognitive load, productivity drain, inconsistent methodology documentation

#### Problem 3: Trust Barriers & IP Risks
- Uploading proprietary data to AI assistants (ChatGPT, Claude, Gemini) risks IP leakage to model training pipelines
- No existing AI research tool provides **verifiable provenance logging** for systematic reviews
- Researchers cannot demonstrate their AI-assisted methodology is auditable or reproducible
- In regulated industries (pharma, defense, finance), this is a **compliance showstopper**
- **Impact:** Researchers either avoid AI tools entirely or use them unsafely

### The Market Gap

> No existing solution combines end-to-end research workflow support + IP compliance + private model training potential + provenance-auditable outputs.

## 3. Product Vision

### Core Philosophy

> *"Rigorous research, responsibly delivered."*

Redac is built on the conviction that AI should **augment human research capability** without compromising the integrity, IP rights, or methodological rigor that define trustworthy scholarship. Every design decision flows from four foundational principles:

### Key Principles

1. **Privacy by Design** — Private data is encrypted, isolated, and never exposed to central training pipelines. The architecture enforces this at the infrastructure level, not just by policy.

2. **IP Compliance as Default** — The central corpus contains only public metadata with full provenance logging. Full-text acquisition is tracked per genesis event. Users retain all rights to their private corpora.

3. **Radical Transparency** — Every AI-generated output includes a complete methodology log: what sources were searched, what agents ran, what decisions were made, and with what confidence.

4. **Universal Access** — A generous free tier ensures that researchers at resource-constrained institutions (global south universities, independent researchers) have access to world-class research tooling.

## 5. Feature Specifications

### 5.1 Core Capabilities (Phase 1)

#### AI-Driven Discovery Engine
- **Natural language queries:** "Find papers on federated learning in healthcare published after 2020 with clinical trial data"
- **Multi-source search:** Crossref, OpenAlex, Semantic Scholar, arXiv, PubMed (via public APIs)
- **Snowballing:** Automated forward/backward citation tracing from seed papers
- **Semantic similarity search:** Vector-based paper recommendation using pgvector
- **Filtering:** Date range, methodology, publication type, citation count, journal tier

#### Intelligent Screening & Extraction
- **Abstract screening:** AI agent pre-screens papers against inclusion/exclusion criteria
- **Full-text extraction:** Structured data extraction from key findings, methods, sample sizes
- **Contradiction detection:** Flags papers with conflicting findings
- **Confidence scoring:** Every extraction tagged with confidence level and source reference

#### Dual-Corpus Architecture
- **Central Public Corpus:**
  - Contains public metadata only (titles, abstracts, DOIs, author affiliations)
  - Enriched via Crossref/OpenAlex APIs
  - Shared across all users (read-only)
  - Full provenance logging for all additions
  - Target: 10K+ papers at launch, growing to 1M+ by Year 2

- **Private Encrypted Corpus (per user):**
  - Full-text documents, notes, annotations, unpublished manuscripts
  - AES-256 encryption at rest; per-user encryption keys
  - Never indexed in central corpus
  - Searchable only by the owning user
  - Import via PDF upload, URL import, or API

#### Output Generation
- **Literature Review:** Structured academic lit review in user-specified style (APA, Vancouver, Chicago)
- **Research Summary Slides:** Auto-generated PowerPoint/Google Slides from synthesis
- **Evidence Timeline:** Chronological visualization of research evolution on a topic
- **PRISMA Diagram:** Auto-generated flow diagram for systematic reviews
- **Gap Analysis Report:** AI-identified research gaps and future directions

#### Transparent Methodology Logging
- Every search query logged with timestamp and parameters
- Every screening decision recorded with AI rationale
- Complete agent execution trace downloadable as PDF
- Supports PRISMA 2020 and PROSPERO registration documentation

#### Subscription & Token Model
- **Free Tier:** 50 tokens/month, 100 paper corpus limit, 1 output/month
- **Pro ($20/month):** 500 tokens/month, unlimited corpus, 10 outputs/month
- **Team ($49/user/month):** 2,000 tokens/user/month, shared workspace, priority processing
- **Tokens:** 1 token ≈ 1 AI agent task (search, screen batch, extract, generate section)

## 7. Security & Privacy Architecture

### Data Isolation Model

```
┌─────────────────────────────────────────────────────────────┐
│                     Redac Platform                          │
│  ┌─────────────────────┐    ┌───────────────────────────┐  │
│  │  Central Public     │    │  Private Corpus           │  │
│  │  Corpus             │    │  (Per User)               │  │
│  │  • Public metadata  │    │  • AES-256 encrypted      │  │
│  │  • Provenance logs  │    │  • User-only key          │  │
│  │  • Shared (R/O)     │    │  • Never shared           │  │
│  └─────────────────────┘    └───────────────────────────┘  │
│           ↓ No cross-contamination enforced at DB level      │
└─────────────────────────────────────────────────────────────┘
```

### Security Controls

| Layer | Control |
|-------|---------|
| **Transport** | TLS 1.3 for all communications |
| **Storage** | AES-256 encryption at rest for private corpora |
| **Authentication** | JWT with short expiry + refresh token rotation |
| **Authorization** | Row-level security in PostgreSQL; per-user corpus isolation |
| **Secrets Management** | Environment variables; never in config files |
| **API Security** | Rate limiting, input validation, OWASP Top 10 mitigations |
| **Audit Logging** | Immutable access logs for compliance |

### Privacy Guarantees

1. **Private data never leaves user control** — no cross-user exposure at infrastructure level
2. **No training on private data** — private corpus content never enters central model training
3. **Opt-in only for Phase 3** — federated learning is strictly voluntary with blockchain-managed consent
4. **Right to deletion** — full corpus deletion on account closure with cryptographic verification

### Compliance Approach

- **GDPR:** Data minimization, right to access/deletion, DPA agreements for EU users
- **CCPA:** Data disclosure and deletion rights for California residents
- **FERPA:** Considerations for academic institution partnerships
- **HIPAA:** Phase 3 enterprise tier will pursue BAA for healthcare R&D teams
- **Regular Security Audits:** Quarterly penetration testing pre-launch; annual third-party audits

| Component | Technology | Rationale |
|-----------|-----------|-----------|

| **Task Decomposition** | Cortex Agent (OpenClaw) | Breaks complex research tasks into executable sub-tasks |
| **Model Abstraction** | LiteLLM | Unified interface to OpenAI, Anthropic, local models |
